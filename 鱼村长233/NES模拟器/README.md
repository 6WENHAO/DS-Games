# HTML5 NES 模拟器

纯 **HTML + CSS + JavaScript** 编写的红白机（NES / FC）模拟器，零依赖、无构建工具，浏览器直接运行。项目同时是一份"如何写 NES 模拟器"的教学示例：核心模块与浏览器完全解耦，每个文件对应一块真实硬件，代码内有详细的原理注释。

## 功能

- 6502 CPU：全部官方指令 + 常用非官方指令，周期精确（含跨页/分支惩罚）
- 扫描线级 PPU：滚动（loopy 寄存器）、精灵、sprite 0 hit、8×16 精灵、左列裁剪、灰度
- APU：方波 ×2 / 三角波 / 噪声 / DMC 五通道，非线性混音，Web Audio 输出
- Mapper：0 (NROM)、1 (MMC1)、2 (UxROM)、3 (CNROM)、4 (MMC3 含扫描线 IRQ)、7 (AxROM)、11、66
- 外设：键盘 + 手柄（Gamepad API）、电池存档（localStorage）、截图、全屏、拖放加载

## 快速开始

直接双击打开 `index.html` 即可（无需服务器）；或启动任意静态服务器：

```bash
npx serve .
# 或
python -m http.server
```

然后点击「打开 ROM」或把 `.nes` 文件拖进窗口。本项目不包含任何游戏 ROM，请使用拥有合法授权的文件，或使用免费的自制（homebrew）游戏与测试 ROM。

## 操作

# NES 模拟器原理

## 1. 整体架构

NES 里有两颗各自独立运行的芯片，通过内存映射寄存器沟通：

```
┌──────────────────────────┐         ┌─────────────────────────┐
│  CPU 2A03 @1.79MHz       │         │  PPU 2C02 @5.37MHz      │
│  ├─ 6502 核心            │◄──NMI───┤  256×240 @60Hz          │
│  ├─ APU（音频）          │──$2000-7─►  8KB 图案 + 2KB 显存    │
│  └─ 2KB RAM              │         │  256B 精灵表 + 调色板    │
└───────────┬──────────────┘         └───────────┬─────────────┘
            │ CPU 总线                            │ PPU 总线
┌───────────┴──────────────────────────────────── ┴─────────────┐
│  卡带：PRG ROM（程序）── CHR ROM/RAM（图块）── Mapper（银行切换）│
└────────────────────────────────────────────────────────────────┘
```

**同步是模拟器的第一要务**：1 个 CPU 周期 = 恰好 3 个 PPU 点（NTSC）。本项目在 `nes.js runFrame()` 中以"一条 CPU 指令"为步长推进：

```js
const cycles = this.cpu.step();   // 执行一条指令，返回周期数 n
while (dots-- > 0) this.ppu.tick(); // PPU 前进 3n 个点
this.apu.step(cycles);            // APU 前进 n 个周期
```

一帧约 29780 个 CPU 周期 / 89342 个 PPU 点；PPU 扫到第 241 线进入 VBlank 时本帧结束。

## 2. CPU（js/cpu.js）

6502 是 8 位 CPU：3 个数据寄存器（A/X/Y）、栈指针、程序计数器和 8 个标志位。模拟的核心是一张 **256 项操作码表**，每项记录"做什么（指令）、操作数在哪（13 种寻址方式）、花多久（周期数）"。`step()` 的流程：

1. 检查中断：DMA 停顿 → NMI（PPU 的 VBlank 通知，不可屏蔽）→ IRQ（APU/Mapper，受 I 标志屏蔽）
2. 取指令，按寻址方式算出有效地址（注意零页回绕、`JMP ($xxFF)` 的换页 bug 等硬件怪癖）
3. 执行并返回周期数（跨页读 +1，分支命中 +1、再跨页又 +1）

周期数必须准确——游戏靠数周期与 PPU 对齐做画面特效，差几个周期就会闪屏。

## 3. PPU（js/ppu.js）—— 最复杂的部分

### 3.1 帧的时间结构

一帧 = **262 条扫描线 × 每线 341 点**，每点对应一个像素时间：

| 扫描线 | 作用 |
| --- | --- |
| 0-239 | 逐线绘制可见画面（256×240） |
| 240 | 空闲 |
| 241 (第 1 点) | 置 VBlank 标志，触发 NMI → 游戏在此更新显存 |
| 242-260 | VBlank 持续 |
| 261 | 预渲染线：清标志、恢复滚动位置、奇数帧少 1 点 |

### 3.2 图形数据如何变成像素

三层查表（都是索引，没有任何位图）：

```
命名表(选哪个图块) ──► 图案表(图块的两个位平面 → 0-3 号色)
属性表(选哪组调色板) ─┘            │
                                   ▼
                调色板 RAM（组内 0-3 → 64 色主调色板索引）──► RGB
```

- **图案表**：每个 8×8 图块 16 字节 = 两个位平面，两位拼出 4 种颜色（0 = 透明）
- **命名表**：32×30 的图块索引矩阵（对应一屏），尾部 64 字节属性表以 16×16 像素为单位选调色板
- **调色板**：背景/精灵各 4 组 ×3 色 + 1 个公共背景色

### 3.3 滚动：loopy 寄存器 v/t/x/w

PPU 用一个 15 位地址寄存器 `v` 同时编码"正在取哪个图块"和"滚动到了哪里"：

```
 yyy NN YYYYY XXXXX
 │   │  │     └─ coarse X（图块列 0-31）
 │   │  └─────── coarse Y（图块行 0-29）
 │   └────────── 命名表选择（2 位，即 4 屏中的哪一屏）
 └────────────── fine Y（图块内的行 0-7）
```

游戏写 `$2005/$2006` 只改暂存的 `t`；硬件在固定时刻把 `t` 复制进 `v`（每线第 257 点复制水平位、预渲染线复制垂直位）。**分屏滚动**（如状态栏固定、场景滚动）正是趁扫描中途改 `t/v` 实现的——所以模拟器必须按点精确模拟这些复制时机，这也是本项目虽按整行渲染、却仍逐点走 `tick()` 的原因。

### 3.4 精灵与 sprite 0 hit

256 字节 OAM 存 64 个精灵（Y、图块、属性、X 各 1 字节）。每线最多 8 个，编号小者优先。**sprite 0 hit** 是重要的同步机制：0 号精灵的不透明像素与背景不透明像素首次重叠时置位 `$2002` bit6，游戏轮询它得知"扫描线到了某个位置"，再切换滚动值——超级马里奥的状态栏就靠它。模拟时在整行渲染阶段算出命中的像素 X，再由 `tick()` 在对应点位真正置位，保证轮询时序正确。

## 4. APU（js/apu.js)

5 个通道各是一台小状态机：**分频器**（决定音高）+ **音量包络**（衰减）+ **长度/线性计数器**（音符时长）+ **扫频**（滑音）。帧计数器以 240Hz/120Hz 驱动这些计数器。

模拟输出的流程：每个 CPU 周期推进通道 → 每约 40.6 个周期（1789773 ÷ 44100）抽取一个样本 → 按 NESdev 的非线性公式混音：

```
pulse_out = 95.88 / (8128/(p1+p2) + 100)
tnd_out   = 159.79 / (1/(t/8227 + n/12241 + d/22638) + 100)
```

→ 一阶高通滤波去直流 → 环形缓冲 → Web Audio 回调消费。模拟速度驱动音频（而不是反过来），缓冲欠载时让尾音指数衰减避免爆音。

## 5. Mapper（js/mappers.js）

CPU 给卡带的窗口只有 32KB、PPU 只有 8KB，大游戏必须**银行切换**：往 ROM 地址写数据（ROM 本不可写），卡带上的逻辑电路把它解释为指令，切换窗口里可见的 ROM 区块。例如 MMC1 要求向同一地址连写 5 次、每次 1 位（串行协议）；MMC3 则有 8 个银行寄存器外加一个**扫描线计数器**，能在指定扫描线向 CPU 发 IRQ（超马 3 的分屏）。

Mapper 还决定**命名表镜像**——2KB 显存如何折叠成 4 个逻辑命名表（水平/垂直/单屏/四屏），直接影响卷轴方向。

## 6. 主循环与视听输出（js/main.js）

- **画面**：PPU 直接把 ABGR 像素写进 canvas `ImageData` 的底层内存（`Uint32Array` 共享视图），每帧一次 `putImageData`，零拷贝；CSS `image-rendering: pixelated` 无模糊放大
- **时序**：`requestAnimationFrame` + 时间累加器锁定 60.0988 FPS，与显示器刷新率解耦；后台切回时最多补 4 帧防雪崩
- **输入**：键盘/手柄状态每帧合并写入手柄移位寄存器（controller.js）
- **存档**：带电池的卡带（`$6000-$7FFF`）定期序列化为 base64 存入 localStorage，按 ROM 哈希区分

## 7. 取舍与已知限制

- 扫描线级渲染（整行一次画完）而非逐点渲染：性能好、兼容绝大多数游戏，但单行中途换调色板/图案表的极端光栅特技会有偏差
- MMC3 的 IRQ 用"每线第 260 点"近似，未模拟 PPU A12 地址线的精确跳变
- DMC 取样不占用（"偷"）CPU 周期；未实现色彩强调位着色、NTSC 滤镜；仅 NTSC 时序
- 不支持的 Mapper 会明确报错提示编号

## 8. 测试

除 `main.js` 外的模块不依赖浏览器 API，可在 Node 中直接加载做冒烟测试（构造内存中的 iNES ROM，验证 CPU 指令结果、PPU 渲染像素值、sprite 0 hit、APU 出样、各 Mapper 读写与 MMC3 IRQ）。
